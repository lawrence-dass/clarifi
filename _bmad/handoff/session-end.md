# Session End — Handover Instructions

Save session state so any agent can resume seamlessly in a new window.

---

## Step 1: Detect Mode

Run:
```bash
git branch --show-current
git status --short
git log -1 --format="%h %s"
```

**Story discovery — try in this order, stop at first match:**

1. Read `_bmad-output/implementation-artifacts/sprint-status.yaml`. Find the first entry matching pattern `number-number-name` with status `in-progress`. If found → use this story.
2. If sprint-status.yaml is missing, malformed, or has no `in-progress` entry → search `_bmad-output/implementation-artifacts/*.md` for a story file containing `Status: in-progress`. Use the first match.
3. If still no match → ask the user: "Which story file are you working on? Provide the path, or reply 'none' for a general handover."

If a story file is identified, read it completely from `_bmad-output/implementation-artifacts/{story-key}.md`.

**Set mode:**
- Unchecked `[ ]` tasks remain in story file → **Mode 1: Mid-story**
- All tasks checked `[x]` in story file → **Mode 2: Story complete**
- User replied "none" or no story found after asking → **Mode 2: General handover** (skip story file update)

---

## Step 2: Draft Updates

### Mode 1 — Mid-story

**Dev Agent Record heading compatibility:**
- Before drafting story-file changes, inspect the existing headings in the story file's Dev Agent Record.
- Use the existing heading names exactly as they appear.
- Treat `### Debug Log References` and `### Debug Log` as equivalent debug-log sections.
- Treat `### Completion Notes List` and `### Completion Notes` as equivalent completion-notes sections.
- Do not create duplicate Debug Log or Completion Notes sections. Only create a missing section if no equivalent section exists.

**Story file Dev Agent Record draft** (DO NOT change Status or checkboxes):
```
### Agent Model Used
{agent name and model version}

### {existing debug-log heading}
{any blockers, errors, or issues hit this session — or "None"}

### {existing completion-notes heading}
Resume: {exact next action — subtask name, file path, what approach to take}
Uncommitted: {list of modified files from git status, or "None"}
Session: {1-2 sentences on what was done and what remains}
```

**CURRENT.md draft:**
```markdown
# Handover — {YYYY-MM-DD HH:MM} | {agent name + model}

## Mode
Mid-story (context pressure)

## Active Story
- **Story**: {story-id} — {story-title}
- **Branch**: {branch-name}
- **Status**: in-progress
- **Progress**: {X of Y tasks complete}
- **Risk tier**: {1/2/3 if recorded in the story file, else "unset"}

## Resume Point
{Exact next action written for a stranger reading cold.
Include: which subtask, which file, what approach, why.}

## Uncommitted Files
{list from git status, or "None — working tree clean"}

## What Happened This Session
- {bullet}
- {bullet}

## Decisions Made
{Decisions that would be re-debated or re-made without this note. Be specific.}

## Last Workflow
{Last BMAD skill or command run this session}

## Context Needed
{Anything the next agent must load or know that isn't in the story file — specific architecture shards, guardrails in play, external constraints, WIP assumptions}

## Next Action
Open the story file and run the `bmad-dev-story` skill on it.

## References
- Story: `_bmad-output/implementation-artifacts/{story-key}.md`
- Sprint: `_bmad-output/implementation-artifacts/sprint-status.yaml`

---
*Generated by /session-end — {YYYY-MM-DD HH:MM}*
```

### Mode 2 — Story complete

**Story file updates:**
- Set `Status: review` in the story file
- Update `### File List` with all files created or modified
- Update the existing completion-notes section (`### Completion Notes List` or `### Completion Notes`, whichever appears in the story file):

```
{What was built, key decisions, anything the next story should know}
Agent: {agent name and model version}
```

**sprint-status.yaml update:**
- Set `development_status[{story-key}]: review` — this is the authoritative source; the story file update above must be consistent with it

**CURRENT.md draft:**
```markdown
# Handover — {YYYY-MM-DD HH:MM} | {agent name + model}

## Mode
Story complete

## Completed Story

- **Branch**: {branch-name}

- {bullet}
```

## Step 3: Validate Handoff Quality

Before showing drafts, verify the following. If any check fails, fix the draft before proceeding.

| Check | Requirement |
|-------|-------------|
| Focus is specific | One sentence, names the exact story and subtask — not "working on the project" |
| Resume Point (Mode 1) | Names a specific file, subtask, and approach — not "continue implementation" |
| Next Action | A concrete skill/command or instruction — not "keep working" |
| Reference files exist | Run a quick check that referenced paths actually exist in the repo |
| No duplication | CURRENT.md references story file and sprint-status by path — does not copy their content |
| No code snippets | File paths and function names are fine; full code blocks are not |
| Length | CURRENT.md body under 50 lines (templates and headers excluded) |

---

## Step 4: Show Drafts and Confirm

Present both drafts clearly — the story file section and the CURRENT.md — in a single message. Ask:

> "Approve writing these updates? Reply **yes**, **no**, or describe what to change."

- **yes** → proceed to Step 5
- **no** → discard and exit
- anything else → treat as edit instructions, revise, re-show, ask again

---

## Step 5: Collect Cross-Story Learnings (Mode 2 only)

Before writing any files, ask:

> "Any cross-story learnings to add to `project-context.md`? These will be loaded automatically by all future stories. (yes / no)"

If **yes**:
- Ask: "What is the learning?"
- Draft a concise addition to `_bmad-output/project-context.md` under a dated heading: `## Learnings from Story {story-id} — {YYYY-MM-DD}`
- If `_bmad-output/project-context.md` does not exist yet, create it with a one-line title (`# Clarifi — Cross-Story Learnings`) before appending
- Show the draft and confirm with the user
- On approval, add `- project-context.md: _bmad-output/project-context.md` to the **References** section of the CURRENT.md draft before writing it

If **no**, proceed to Step 6 immediately.

---

## Step 6: Write Files

On confirmation (from Step 4), write in this order:

1. **Story file** — update Dev Agent Record section (and Status + File List if Mode 2). Only modify those sections; leave everything else untouched.
2. **`sprint-status.yaml`** — if Mode 2 (story complete), set `development_status[{story-key}]: review`. Skip for Mode 1.
3. **`_bmad-output/CURRENT.md`** — write the approved content (including any References additions from Step 5).
4. **Timestamped copy** — write identical content to `_bmad-output/handoff/{YYYY-MM-DD-HH-MM}.md`. Create the `handoff/` directory if it does not exist.
5. **`_bmad-output/project-context.md`** — if learnings were approved in Step 5, append (or create then append) the new section now.

---

## Step 7: Confirm Complete

Display:

```
✅ Handover saved

  CURRENT.md              → _bmad-output/CURRENT.md
  Timestamped copy        → _bmad-output/handoff/{timestamp}.md
  Story Dev Agent Record  → updated
  {if Mode 2} Story status        → review
  {if Mode 2} sprint-status.yaml  → development_status[{story-key}]: review
  {if project-context updated} project-context.md → updated

Safe to close this window.
```

## Handoff Quality Rules

### Must Include

| Field | Why Critical |
|-------|--------------|
| Current Focus / Mode | What to resume working on |
| Active Story | Prevents starting wrong work |
| Decisions Made | Prevents re-debating settled issues |
| Next Action | Eliminates "where was I?" |
| References | Ensures right files are loaded |

### Keep Lean

- Reference BMAD artifacts by path — do not copy their content
- Story details → point to the story file
- Architecture decisions → point to architecture docs
- Sprint state → point to sprint-status.yaml

### Avoid

- Full code snippets (use file path:line references instead)
- Duplicating sprint-status.yaml content
- Verbose explanations — be precise, not thorough
- Historical context not needed for the next session

---

## Error Handling

| Scenario | Action |
|----------|--------|
| `sprint-status.yaml` missing | Note "Planning phase — no sprint tracking" and derive story state from story file directly |
| No `in-progress` story found | Note "Between stories" and set Mode 2 (General handover) |
| Story file missing | Note missing path in handover, skip story file update |
| Git unavailable | Skip git state section, note "Git unavailable" |

---

## Notes

- Write Resume Point (Mode 1) and Decisions Made as if the next reader has never seen this codebase
- Do not redact file paths, function names, or technical detail — specificity is the point
- If sprint-status.yaml is missing, derive story state from the story file Status field directly
- Code review is manual in this repo — never auto-start it; only suggest it
